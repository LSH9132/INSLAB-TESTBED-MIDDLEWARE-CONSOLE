# 실험 환경구성

## 1. 물리 환경

- 스위치: NexTU L2 관리형 스위치
    - 포트 1개당 Pi 1대 연결
    - VLAN Tagged 설정 필수 (미설정 시 링크 불통, 정전 후 초기화 주의)
- 실험 노드: Raspberry Pi 14대 (pi1 ~ pi14)
- 연결: 각 Pi -> 스위치 1포트 (1노드 = 1포트)

---

## 2. 네트워크 구조

### OVS 브릿지 구조

```
eth0 (IP 없음)
  └── mgmt-br (10.10.0.x/24)   # 관리망: SSH, 모니터링용
        |
   patch-m-t <-> patch-t-m     # VLAN 트렁크 (patch port pair)
        |
     test2-br                  # 실험망 브릿지
        ├── int12 (tag=12, 10.1.2.x/30)
        ├── int14 (tag=14, 10.1.4.x/30)
        └── ...                # 각 링크별 int* 인터페이스
```

- **mgmt-br**: eth0 물리 포트 연결, 관리망 IP 할당. 실험과 무관하게 항상 유지
- **test2-br**: 실험용 VLAN 인터페이스 모음. 브로드캐스트가 VLAN 단위로 격리됨
- **patch-m-t / patch-t-m**: 두 브릿지를 연결하는 가상 케이블. Trunk로 설정해 지정 VLAN 프레임만 통과

### IP 체계

- 관리망: `10.10.0.x/24` (mgmt-br에 할당)
- 실험망: `10.{낮은노드}.{높은노드}.{1 or 2}/30`
    - 낮은 번호 노드 -> `.1`, 높은 번호 노드 -> `.2`
- 인터페이스 명명: `int` + VLAN 번호
    - 노드 번호 10 이상은 한 자리 변환: 10->0, 11->1, 12->2, 13->3, 14->4
    - 양쪽 모두 10 이상: 역순 (pi10-pi11 -> VLAN 10, pi11-pi14 -> VLAN 41)
    - 한쪽만 10 이상: 순서 유지 (pi7-pi10 -> VLAN 70, pi8-pi11 -> VLAN 81)

### 이 구조를 선택한 이유

- 물리 스위치 1대에 모든 Pi가 연결 -> Pi 간 직접 케이블 연결 불가
- VLAN으로 링크별 브로드캐스트 도메인을 격리해 멀티-홉 Ad-Hoc 실험 가능
- Babel 등 메쉬 라우팅 프로토콜이 단일 L2 도메인으로 동작하지 않도록 분리

### 장점

- 토폴로지 변경 시 케이블 없이 인터페이스 추가/삭제만으로 가능
- 실험 중 링크 Down/Up 해도 관리망(SSH) 유지
- tc/OVS QoS 명령으로 지연, 손실, 대역폭 제한 삽입 가능

## 4. Babel 라우팅 프로토콜

- 거리-벡터 기반 + 링크-상태 정보가 결합된 하이브리드 메쉬 라우팅 프로토콜
- 1~2초 단위 Hello 교환으로 빠른 수렴
- RTT, 손실률 기반 적응형 메트릭
- 경량 설계 -> Raspberry Pi 환경에 적합
- **OVS 설정이 완전히 완료된 후에 설정할 것**

### /etc/babeld.conf (pi1 예시)

```bash
# Babeld configuration - auto-generated by pi_setup
interface int12
interface int14
interface int15
```

### /etc/systemd/system/babeld.service

```
[Unit]
Description=Babel routing daemon
Wants=network-online.target
After=network-online.target

[Service]
Type=simple
ExecStart=/usr/sbin/babeld -D -c /etc/babeld.conf -I /run/babeld.pid
Restart=on-failure
RestartSec=2

[Install]
WantedBy=multi-user.target
```

### 주요 명령어

```bash
# 설치
sudo apt update && sudo apt install -y babeld

# 서비스 등록 및 시작
sudo systemctl daemon-reload
sudo systemctl enable babeld
sudo systemctl start babeld

# 재시작
sudo systemctl restart babeld

# 경로 확인
traceroute -n -I 10.0.41.2
```

## 5. OVS 주요 명령어

```bash
# 패치 포트 생성 (브릿지 간 연결)
sudo ovs-vsctl add-port mgmt-br patch-m-t -- set interface patch-m-t type=patch options:peer=patch-t-m
sudo ovs-vsctl add-port test2-br patch-t-m -- set interface patch-t-m type=patch options:peer=patch-m-t

# 트렁크 VLAN 설정
sudo ovs-vsctl set port patch-m-t trunks=12,14,15
sudo ovs-vsctl set port patch-t-m trunks=12,14,15

# 실험 인터페이스 생성 (int12 예시)
sudo ovs-vsctl add-port test2-br int12 -- set interface int12 type=internal
sudo ovs-vsctl set port int12 tag=12
sudo ip addr add 10.1.2.1/30 dev int12
sudo ip link set int12 up

# 169.254.x.x 링크로컬 주소 제거
ip -o -4 addr show | awk '$2 ~ /^int/ && $4 ~ /^169\.254/ {print $4, $2}' | while read -r ip dev; do sudo ip addr del "$ip" dev "$dev"; done

# 라우팅 캐시 초기화
sudo ip route flush cache
```

## 6. pi_setup/ 스크립트 설명

테스트베드 자동 설정을 위한 Python 스크립트 모음. Windows에서 파일을 배포하고 각 Pi에서 실행한다.

### [config.py](http://config.py)

- 역할: 14대 Pi의 토폴로지 정보를 담은 중앙 설정 파일
- 주요 내용
    - `MGMT_IP`: Pi 번호 -> 관리망 IP 매핑
    - `LINKS`: 27개 링크 정의 (node_a, node_b, vlan, 인터페이스명, ip_a, ip_b)
    - `get_pi_config(pi_num)`: 특정 Pi의 인터페이스/IP/트렁크 목록 반환

### [setup.py](http://setup.py)

- 역할: 각 Pi에서 실행하는 OVS 자동 설정/검증 스크립트
- 동작 순서
    1. hostname에서 Pi 번호 자동 감지 (또는 `--pi N`)
    2. [config.py](http://config.py)에서 기대 설정값 로드
    3. 현재 OVS 상태 수집 (포트, IP, VLAN, 트렁크)
    4. 불일치 항목 자동 수정
        - eth0 IP 제거 후 mgmt-br 연결
        - 관리 IP 할당, 잘못된 IP 제거
        - int* 인터페이스 생성/삭제, VLAN 태그, IP 할당
        - 169.254.x.x 링크로컬 주소 제거
        - 패치포트/트렁크 설정, 충돌 브릿지 IP 제거
        - babeld.conf 재생성, dhcpcd.conf 업데이트, babeld 재시작
    5. 인접 노드 ping으로 검증
- 옵션: `--dry-run` (예정 동작만 출력), `--status` (현재 상태만 표시)

### babel_[setup.py](http://setup.py)

- 역할: babeld 설치 및 서비스 설정 전용 스크립트
- 미설치 시 apt 또는 오프라인 패키지(`packages/babeld_1.12.1-1_arm64.deb`) 설치
- babeld.conf 자동 생성, babeld.service 등록 및 시작

### check_[status.py](http://status.py)

- 역할: 각 Pi 상태 종합 진단 (읽기 전용)
- 진단 항목: OVS 브릿지, 관리망 IP, 트렁크, 인터페이스 VLAN/IP/상태, 인접 ping, VLAN 프레임 수신, Babel 라우팅, dhcpcd 설정

### check_[dhcp.py](http://dhcp.py)

- 역할: DHCP 누출 진단 및 수정
- dhcpcd / NetworkManager 중복 실행 감지 및 수정 (`--fix`)
- denyinterfaces 설정 확인, 169.254.x.x 주소 감지/제거, 관리망 IP 누출 감지

### check_all_[links.py](http://links.py)

- 역할: Windows에서 전체 14대 Pi, 27개 링크 연결성 일괄 테스트
- paramiko로 각 Pi에 SSH 접속 후 int* 인터페이스로 인접 노드 ping
- 54개 테스트(양방향) 결과를 OK/FAIL/ERR + RTT로 출력

### [send.py](http://send.py)

- 역할: Windows에서 Pi들에게 파일을 SCP로 배포
- paramiko 사용, 비밀번호 자동 입력 (기본값: 1234)

```bash
python send.py setup.py                   # 전체 Pi에 배포
python send.py check_dhcp.py --pi 5 7    # 특정 Pi에만 배포
```
